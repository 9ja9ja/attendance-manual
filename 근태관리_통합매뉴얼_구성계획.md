# 근태관리 통합 매뉴얼 구성계획

## 1. 문서 목적

이 문서는 `근태관리` 서비스 전반을 대상으로 한 HTML 매뉴얼 제작의 기준 문서다.

목표는 다음과 같다.

- 피그마 기획서와 기존 계산/학습 문서를 기반으로 근태관리 전체 업무를 설명하는 통합 매뉴얼을 만든다.
- 기존 `html-manual` 스타일을 따르되, 기존 글로벌 인덱스에는 연결하지 않고 `근태관리` 전용 매뉴얼 세트를 독립적으로 구성한다.
- 향후 화면 추가, 정책 변경, 계산 로직 상세화가 발생하더라도 동일한 구조 안에서 HTML 페이지를 계속 확장할 수 있도록 정보구조를 먼저 확정한다.

## 3. 제작 방향

근태관리 매뉴얼은 `한 개의 긴 HTML`보다 `근태 전용 인덱스 + 업무별 상세 HTML` 구조로 가는 것이 적합하다.

이유는 다음과 같다.

- 근태관리 범위가 `근로제`, `요청/승인`, `직원 근태`, `휴가`, `출장`, `인정근로시간`, `월 마감`으로 분리되어 있다.
- 피그마 화면도 관리자/직원/정책/유형설정/계산다이어그램으로 나뉘어 있어 한 페이지에 모두 넣으면 탐색성이 떨어진다.
- 향후 계산 로직이나 승인 정책이 바뀌면 특정 업무 HTML만 교체하는 방식이 유지보수에 유리하다.

따라서 1차 산출은 아래 두 계층으로 구성한다.

- `근태관리 전용 index.html`
- `업무별 상세 HTML`

## 4. HTML 스타일 적용 원칙

`html-manual/CLAUDE.md`의 기존 스타일을 아래처럼 적용한다.

### 4.1 인덱스 페이지

- 역할: 전체 업무 구조 소개, 페이지 진입점 제공, 탭 또는 카드형 네비게이션 제공
- 적용 스타일: `Template C` 성격 우선
- 특징:
  - 어두운 배경 기반
  - 업무 영역별 탭 또는 카드
  - 전체 흐름도와 빠른 이동 링크 제공

### 4.2 상세 페이지

- 역할: 특정 업무를 실제 사용자/관리자 관점에서 설명
- 적용 스타일: `Template A` 성격 우선
- 특징:
  - 섹션 구분이 명확한 상세 설명형 문서
  - 화면 구성, 업무 목적, 주요 필드, 처리 흐름, 예외/주의사항을 함께 기술
  - 필요 시 표, 상태 흐름, 계산식, 정책 노트를 포함

### 4.3 프로세스 강조 페이지

- 역할: 계산 로직, 승인 흐름, 마감 절차처럼 관계 중심 설명이 필요한 영역
- 적용 스타일: `Template A` 기본 + `Template C`식 흐름 박스 일부 혼합
- 대표 대상:
  - 인정근로시간 / 월마감
  - 요청/승인 상태 흐름

## 5. 근태관리 매뉴얼 정보구조(IA)

### 5.1 최상위 분류

근태관리 통합 매뉴얼은 아래 7개 영역으로 분리한다.

1. 개요
2. 근로제 관리
3. 근태 요청 관리
4. 직원 근태
5. 휴가 관리
6. 출장/외근 유형 관리
7. 인정근로시간 및 월 근태 마감

### 5.2 업무 간 연결 구조

업무는 아래 순서로 이어진다.

1. `근로제 관리`
근무 기준, 근로제 유형, 근무 패턴, 정책을 정의한다.

2. `휴가 / 출장 / 외근 유형 관리`
승인 시 인정 방식을 바꾸는 유형 마스터를 정의한다.

3. `근태 요청 관리`
직원 또는 관리자의 요청/정정/승인 데이터를 확정한다.

4. `직원 근태`
출퇴근, 요청, 이상 상태, 예상 근무 정보를 일 단위로 확인한다.

5. `인정근로시간 및 월마감`
근로제, 근무패턴, 요청승인, 출퇴근, 휴게, 유형설정, 보정데이터를 조합해 마감 기준 데이터를 만든다.

## 8. 작성 규칙

각 HTML 페이지는 아래 규칙을 공통 적용한다.

1. `업무 목적`을 먼저 설명한다.
2. `누가 사용하는 화면인지`를 명확히 적는다.
3. `주요 필드/설정값`과 `실제 업무 의미`를 함께 쓴다.
4. `처리 흐름`과 `다음 업무와의 연결`을 꼭 적는다.
5. 계산/판정에 영향을 주는 설정은 별도 강조 박스로 노출한다.
6. 화면 설명만 하지 말고, `정책 의미`와 `마감 영향`까지 연결한다.

## 13. 생산원가 매뉴얼 구조 대비 현재 근태 구조 진단

생산원가 매뉴얼 구조와 비교해보면, 현재 근태매뉴얼은 `문서 수가 많다`기보다 `상세 페이지를 그대로 좌측 메뉴에 펼쳐 놓은 구조`라는 점이 더 큰 부담이다.

비교하면 아래와 같다.

| 비교 항목 | 생산원가 매뉴얼 | 현재 근태매뉴얼 | 판단 |
|----------|----------------|----------------|------|
| 좌측 메뉴 역할 | 상위 업무 묶음 진입 | 개별 상세 페이지 직접 진입 | 근태 쪽이 탐색 깊이를 바로 노출 |
| 세부 문서 탐색 | 섹션 안 서브탭 중심 | 버튼 다수 직접 노출 | 근태 쪽이 피로도가 높음 |
| 문서 구조 | 상세문서와 Flow를 분리하고, 인덱스는 허브 역할 중심 | 상세문서 분리는 잘 되었지만, 인덱스가 허브보다 목록에 가까움 | 구조는 좋고 진입 설계 보완 필요 |
| 신규 사용자 관점 | 1차 업무군 선택 후 2차 탭 선택 | 처음부터 선택지가 많음 | 근태 쪽이 진입 난이도가 높음 |

정리하면, 현재 근태매뉴얼의 문제는 문서 자체보다 `인덱스 IA`에 가깝다.

## 14. 근태 인덱스 IA 재편 원칙

재편 방향은 아래 4가지를 기본 원칙으로 둔다.

1. 좌측 메뉴는 `업무군`만 노출한다.
2. 상세 문서는 좌측이 아니라 `상단 탭` 또는 `서브탭`으로 노출한다.
3. 이미 만든 상세 HTML은 삭제하지 않고 그대로 유지한다.
4. 인덱스는 `탐색 허브` 역할에 집중하고, 상세 내용은 개별 문서에서 읽게 한다.

즉, 앞으로의 구조는 아래와 같다.

- 좌측 메뉴: 상위 업무군 선택
- 상단 탭: 해당 업무군 안의 상세 문서 선택
- 본문: iframe으로 실제 HTML 로드

## 15. 권장 상위 메뉴 재구성안

좌측 메뉴는 아래 6개로 줄이는 것을 권장한다.

1. `개요`
2. `기준설정`
3. `운영처리`
4. `휴가운영`
5. `마감정산`
6. `참고자료`

이 정도면 생산원가 매뉴얼처럼 상위 구조는 간결하고, 근태 특유의 세부 화면도 유지할 수 있다.

- `참고자료`

이 그룹들은 성격이 비교적 분명하므로 분리 유지가 적절하다.

## 18. 다음 구현 방향

실제 인덱스 개편은 아래 순서로 진행한다.

1. 좌측 메뉴를 상위 6개 그룹으로 재구성한다.
2. 메인 영역 상단에 그룹별 탭 바를 추가한다.
3. 각 탭이 해당 HTML 파일을 iframe으로 로드하도록 연결한다.
4. 기존 HTML 파일은 그대로 두고 탐색 구조만 바꾼다.

이 방향이 생산원가 매뉴얼 구조를 가장 자연스럽게 근태에 맞춰 가져오는 방식이다.
